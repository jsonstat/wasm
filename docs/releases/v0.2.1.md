## jsonstat-wasm v0.2.1

A **bug-fix release**. No API changes; no breaking changes. Drop-in replacement for v0.2.0.

### Fixed

- **`fromObject()` now tolerates `null` role values.** `jsonstat-toolkit` normalizes any dataset it touches and may emit `role.classification = null` (and null out other role keys). When such a mutated object was passed back into the WASM library via `JSONstat.fromObject()` — the object fast path used by `JSONstat(obj)` — `serde-wasm-bindgen` tried to deserialize the `null` as `Vec<String>`, calling `Reflect.get(null, ...)` and throwing `TypeError: Reflect.get called on non-object`. A custom null-tolerant deserializer now coerces any `null` role value to an empty `Vec<String>`, matching how the string path (`serde_json`, which is null-tolerant in `untagged`) already handled it. The string path (`JSONstat(jsonString)`) was never affected.

  This was most visible when interoperating with `jsonstat-toolkit`: passing a toolkit-constructed dataset (or any object that has been through the toolkit's normalizer) into the WASM library would throw. With the fix, round-tripping through the toolkit "just works".

### Verification

- Full host test suite (57 tests) passes.
- A reproduction suite covering every `null` role permutation (`role.classification = null`, `role.metric = null`, multiple nulls, all-null, `role` itself `null`) now parses OK on **both** the object path (`fromObject`) and the string path (`new`), plus the real `oecd.json` sample with `role.classification = null`.
