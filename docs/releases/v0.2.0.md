## jsonstat-wasm v0.2.0

A **performance overhaul** focused on closing the gap (and pulling ahead) of the plain-JS `jsonstat-toolkit`. The motivation: profiling showed the WASM build was not faster than pure JS because the dominant costs were (a) parsing with `serde_json` (slower than V8's native `JSON.parse`), (b) object-input round-tripping through `JSON.stringify`, and (c) a `serde_json::Value` â†’ `serde-wasm-bindgen` double-walk on every value crossing the WASMâ†’JS boundary. v0.2.0 attacks all three.

This release has **one minor breaking change** (the `value` getter may now return a `Float64Array`) â€” hence the minor-version bump. All other API surface is unchanged.

### âš ď¸Ź Breaking change: `value` getter returns a `Float64Array` for all-numeric datasets

When **every** value in a dataset is a number (the overwhelmingly common case), the `value` getter â€” and `Data()` / `Data()`-slice in values-only mode (`include_status = false`) â€” now returns a **`Float64Array`** instead of a plain `Array` of boxed numbers.

- **Index access is unchanged:** `ds.value[i]`, `.length`, iteration, and spread all behave identically.
- **What breaks:** `Array.isArray(ds.value)` returns `false` on all-numeric datasets. Code that type-checks the result should use `Array.from(ds.value)` (or `[...ds.value]`) to normalize it back to an `Array` if needed. Mixed datasets (any string/null value) still return a plain `Array`.
- **Why:** a single contiguous `f64` buffer is dramatically cheaper to produce and consume than `N` separately-boxed JS numbers â€” a near-linear speed-up for large numeric datasets.

### Parsing

- **Object-input fast path (`fromObject`).** `JSONstat(obj)` no longer `JSON.stringify`s the object and re-parses the string with `serde_json`. A new static `JSONstat.fromObject()` deserializes the already-parsed JS object straight into the typed Rust model via `serde-wasm-bindgen`, letting the JS engine's native `JSON.parse` own the lexing and paying only one WASM-side traversal. The facade's object branch uses it automatically.
- **URL path kept optimal.** `JSONstat(url)` still parses the fetched body via `serde_json` on the WASM side (a single pure-Rust traversal with no per-token JS-engine crossings). Routing it through `resp.json()` + `fromObject` would add a traversal and a per-element boundary crossing â€” a regression â€” so it is deliberately unchanged. (`simd-json` was evaluated and rejected: it requires `target-feature=+simd128`, which breaks older browsers, and its scalar fallback is slower than `serde_json`.)

### Output boundary (WASM â†’ JS)

- **Direct JS construction on the hot paths.** `Data()`, `Datum()` and `value()` now build `js_sys::Object` / `js_sys::Array` instances directly instead of constructing a `serde_json::Value` tree and re-walking it with `serde-wasm-bindgen`. This removes one full traversal and the serde trait dispatch per cell. The `DataPoint` helper struct is removed.
- **Per-cell status without cloning.** A new `status_at_js` fast path materializes each cell's status directly as a `JsValue` from the borrowed `serde_json::Value`, so `Data()`'s status-bearing loops no longer clone a `Value` and re-serialize it per cell. (The `Transform()`/`Dice()`/`Unflatten()` paths, which build `Value` containers, keep the original `status_at`.)
- **Cached property keys.** The `"value"` / `"status"` object keys are allocated once in a `thread_local` cache, avoiding a fresh JS string per `{value, status}` object in the bulk loops.

### Build profile

- **Speed over size.** The release profile moved from size-first (`opt-level = "s"`, `lto = true`, `wasm-opt -O`) to speed-first: `opt-level = 3`, `lto = "fat"`, `codegen-units = 1`, `panic = "abort"`, `strip = true`, and `wasm-opt -O4`. The `.wasm` binary grows from ~358 KB to ~428 KB (~20%); this is the explicit trade-off for faster runtime code, consistent with a parsing/query library where throughput matters more than a few KB.

### What's unchanged

- The toolkit-compatible API (`JSONstat()`, `Data()`, `Datum()`, `Dimension()`, `Item()`, `Unflatten()`, `Transform()`, `Dice()`, `ToJSON()`) is unchanged in shape and semantics.
- `Transform()`, `Dice()`, and `Unflatten()` outputs are unaffected by the `Float64Array` change (they return their own array/object shapes).

### Verification

- Full host test suite (57 tests), `clippy -D warnings`, `cargo check --target wasm32-unknown-unknown`, and the release `wasm-pack` build (with `wasm-opt -O4`) all pass.
- A benchmark harness comparing the WASM build against `jsonstat-toolkit` is provided at [`.idea/test/bench.html`](../../.idea/test/bench.html) (open it through a local web server so it can load `/pkg/`).
