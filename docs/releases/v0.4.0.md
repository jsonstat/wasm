## jsonstat-wasm v0.4.0

The release that **flips `JSONstat(obj)` from the slowest phase to a win**, and
makes sparse datasets cheap to parse everywhere. No public API change; output is
byte-identical across all parse routes.

### The headline: `JSONstat(obj)` is now fast

v0.3.0 documented `JSONstat(obj)` as "~2.3–4× slower, irreducible." That was
wrong about the *cause*: the cost was not "the WASM boundary" in general, it was
[`serde_wasm_bindgen::from_value`](../src/lib.rs) walking the `value` array
**one `Reflect::get` per cell** (120 000 crossings for a 120k-cell dataset). The
proof was internal — WASM's own object path ran ~2.5× slower than its own string
path on the same data.

v0.4.0 replaces that per-cell walk with a **typed-array bulk copy**, the same
trick that already made the `value` getter and columnar `Transform` fast:

1. **JS pre-flight** ([`describeValue`](../jsonstat.js)) inspects `value` once on
   the JS side. A numeric dense array is coerced to a `Float64Array`
   (`new Float64Array(arr)`, a single native V8 op); a numeric sparse object is
   split into parallel `Uint32Array` indices + `Float64Array` values. Mixed /
   absent / odd shapes return `null` and take the original serde path.
2. **Rust** ([`_fromObjectFast`](../src/lib.rs)) shallow-copies the object with
   `value` stripped, serde-walks only the small metadata, and bulk-copies the
   typed value array with `Float64Array::to_vec()` — **one boundary crossing +
   one memcpy** instead of 120 000.

**Browser** (Chrome 138, [`.idea/test/bench.html`](../.idea/test/bench.html)) —
`parse(object)`, the phase that lost on *every* dataset in v0.3.0:

| Dataset | v0.3.0 wasm | v0.3.0 ratio | **v0.4.0 wasm** | **v0.4.0 ratio** |
|---|---|---|---|---|
| medium dense `[5×100×100]` | 13.4 ms | 2.53 | **1.1 ms** | **0.20 ✅** |
| large dense `[3×200×200]` | 29.1 ms | 2.20 | **2.1 ms** | **0.17 ✅** |
| large sparse `[3×200×200]` | 58.9 ms | 6.47 | **6.0 ms** | **0.66 ✅** |

Large dense dropped **29.1 → 2.1 ms (≈14×)**; large sparse **58.9 → 6.0 ms
(≈10×)** — both now beat the plain-JS toolkit (was 2.2–6.5× slower). Only the
sub-millisecond tiny datasets (oecd, small) still favor JS: those are dominated
by fixed per-call overhead (async gate + facade `Proxy` + WASM call) that is
independent of value size, and is out of scope for this release.

Mechanistic proof on the **same binary** (Node, serde vs fast, large 120k):
large dense **404.5 → 20.6 ms (19.7×)**, large sparse **505.5 → 15.9 ms
(31.8×)** — see [`bench-fromobject.mjs`](../.idea/test/bench-fromobject.mjs).

### Sparse-storage rework

[`DatasetValue::Sparse`](../src/models.rs) was a `HashMap<String, Cell>` keyed
by decimal-string indices (`"0"`, `"3"`, …). That hashed N string keys on
construction (slow for large sparse datasets) and re-hashed on every read. It is
now [`SparseValues`]: **sorted parallel `Vec<usize>` indices + `Vec<Cell>`**,
read by binary search. Built straight from the serde `MapAccess` (string path)
or from the JS typed arrays (object fast path).

Effect on `parse(string)`, browser — large sparse flipped from a loss to a win:

| Dataset | v0.3.0 wasm | v0.3.0 ratio | **v0.4.0 wasm** | **v0.4.0 ratio** |
|---|---|---|---|---|
| large sparse `[3×200×200]` | 23.9 ms | 1.34 (JS wins) | **9.3 ms** | **0.69 ✅** |

Sparse string parse is now faster than dense on the same shape (9.3 ms vs
11.5 ms) — the reverse of v0.3.0, where sparse was the slower of the two.

Every downstream sparse read (`Data()`, `Transform()`, `value` materialization,
`Dice()`) goes through the accessors, so it all benefits with no per-call-site
change.

### What's unchanged

- **No API change.** `JSONstat(string|object|url)`, `Data()`, `Datum()`,
  `Dimension()`, `Item()`, `Unflatten()`, `Transform()`, `Dice()`, `ToJSON()` —
  identical shape and semantics.
- **Byte-identical output.** `ToJSON()`, `.value`, `Dimension()`, and `Data()`
  are verified equal across the string constructor, the serde `fromObject`, and
  the new `_fromObjectFast` for dense-numeric, numeric-sparse, and mixed-value
  datasets (60 Node checks, all pass).
- `fromObject(js)` still exists and takes the serde path (back-compat + the
  mixed-value fallback). `_fromObjectFast` is underscore-prefixed: an internal
  fast path the facade routes to automatically, not part of the public surface.
- The release build profile (`opt-level = 3`, `lto = "fat"`, `codegen-units = 1`)
  is unchanged.

### Verification

- **62 Rust host tests pass** (was 60; +2 for the sparse rework: parallel-vector
  vs entries equivalence, and post-rework sparse reads).
- **60 Node equivalence checks pass** (`verify-fromobject.mjs`): dense / sparse /
  mixed, comparing `.value`, `.n`, `.size`, `.id`, `.Dimension()`, `.Data()`, and
  full `ToJSON()` across all three parse routes.
- New harnesses:
  [`bench-fromobject.mjs`](../.idea/test/bench-fromobject.mjs) (serde-vs-fast on
  one binary) and [`verify-fromobject.mjs`](../.idea/test/verify-fromobject.mjs).

### Upgrading from v0.3.x

Nothing to do — drop-in. The internal sparse representation changed but no
observable behavior did. If you called `JSONstat.fromObject` directly, it still
works exactly as before (and now shares the serde fallback path with the
fast-path router).
